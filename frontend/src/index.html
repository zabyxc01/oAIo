QUICK START
1.Pick a mode in the LIVE tab — oLLMo (LLM + voice), oAudio (voice pipeline), or comfyui-flex (image gen).
2.Services auto-start. Watch the telemetry strip at the top for VRAM / RAM / GPU usage.
3.Use service UIs directly — Open-WebUI (:3000) for chat, ComfyUI (:8188) for images, etc.
4.Monitor everything in LIVE — benchmark, enforcement, kill log, API traffic.
5.Wire services in CONFIG, build pipelines in M³, link machines in FLEET.
If something goes wrong, hit the red KILL ALL button in the top bar to stop all services and clear modes instantly.
SYSTEM ARCHITECTURE
oAIo is a local AI workstation orchestrator. It manages Docker containers running AI services (LLMs, TTS, voice conversion, image generation) on a single GPU, enforcing VRAM budgets so services don’t OOM each other.
  Browser (:9000)
     |
     +-- oAIo API (FastAPI, port 9000) ----+-- oAudio API (port 8002)
     |       |                              |       |
     |   +---+---+---+---+---+              |   Kokoro-TTS --> RVC Proxy
     |   |   |   |   |   |   |              |   F5-TTS, StyleTTS2
     |  Docker SDK  sysfs  config/           |
     |   |                                   |
     +---+--- oaio-net (bridge) ------------+
         |       |        |        |
      Ollama  Open-WebUI  ComfyUI  ...
      (:11434) (:3000)    (:8188)

  /mnt/oaio/*  <-- 17 symlinks -->  physical disks (NVMe/SATA/RAM)
The oAIo container runs inside Docker alongside all services on a shared oaio-net bridge network. It has access to the Docker socket (to manage containers), GPU sysfs (to read VRAM), and all /mnt/oaio/* symlinks (to route storage). The frontend is served as static files from the same port 9000.
LIVE TAB
MODE SELECT
Three built-in modes (oLLMo, oAudio, comfyui-flex) appear as clickable pills. Activating one starts its services and stops any displaced services from the previous mode. Only one mode active at a time — they share a single GPU.
SERVICES GRID
Shows all registered services with live status (running / stopped / error). Click a service row to start or stop it. Color-coded by group: blue = oLLM, amber = oAudio, green = Render, gray = Control.
ENFORCEMENT
Master toggle for automatic OOM protection. Shows warn threshold (default 85%), hard threshold (default 95%), and VRAM ceiling override. When VRAM exceeds the hard threshold, the enforcer kills the lowest-priority service. Priority 1–50, where 1 = last to be killed.
BENCHMARK
Run a quick inference test against Ollama to measure tokens/sec. Results display in a card with model name, tokens generated, and throughput. Requires Ollama to be running with at least one model loaded.
KILL LOG
Rolling log of enforcer actions — which services were warned or killed, timestamps, VRAM levels at time of action. Persists across mode switches.
TELEMETRY STRIP
Top bar showing live gauges: VRAM used/total, RAM used/total, GPU busy %, active mode name. Toggle visibility with the TELEMETRY pill. Updates every 5 seconds.
CONFIG TAB
NODE CANVAS
A full LiteGraph.js node editor. Each registered service appears as a draggable node with I/O ports. Nodes are color-coded by group (bg tint) and status (title bar: green = running, red = stopped). Click a node’s title bar to start/stop the service.
CANVAS CONTROLS
Drag to pan, scroll to zoom. Right-click for context menu. Nodes can be wired together by dragging from an output port to an input port. The graph state persists in the browser.
MODE GROUP BOXES
Each mode draws a colored backdrop behind its service nodes — magenta for oLLMo, cyan for oAudio, yellow for comfyui-flex. Helps visualize which nodes belong to which mode.
BELOW-CANVAS PANELS
Add Service form, Pull Model (downloads Ollama models), SCAN DOCKER / SCAN API / AUTO WIRE buttons for integrating new services. Connection map shows how services route to each other.
ADD SERVICE
Register a new Docker container as an oAIo service. Provide: name, container name, port, group, VRAM/RAM estimates, priority, memory mode, and description. The container must already exist on oaio-net.
M³ TAB — MULTI-MODEL MODE
WHAT IS M³?
M³ chains multiple AI models into inference pipelines and manages training jobs. It is separate from Modes (which control VRAM budgets) and Fleet (which orchestrates remote machines). M³ operates within a single machine.
PIPELINES
A pipeline is a sequence of steps: Input → Model A → Transform → Model B → Output. Each step can be an Ollama model (with temperature, system prompt) or a transform (template substitution, JSON extraction, summarization). Create pipelines in the M³ tab, wire them visually in CONFIG using M³ nodes.
PIPELINE RUNNER
Type input text and run it through a saved pipeline. Each step’s output feeds into the next via {{input}} template substitution. Results display step-by-step with timing. Useful for multi-model reasoning chains.
TRAINING JOBS
Submit fine-tuning jobs with method (QLoRA, LoRA, Full), base model, dataset path, epochs, learning rate, batch size, and rank. Jobs are tracked with status and progress. Training currently requires a separate training container (Ollama is inference-only).
ADAPTERS
Lists completed LoRA/QLoRA adapters from training jobs. Future: convert adapters to GGUF for direct Ollama import.
M³ NODE TYPES (CONFIG CANVAS)
Six LiteGraph nodes: m3-model (Ollama model selector), m3-input (pipeline entry), m3-output (result display), m3-transform (template/extract/summarize/append), m3-router (conditional branching), m3-training (training job with progress bar).
FLEET TAB — MULTI-NODE
WHAT IS FLEET?
Fleet links multiple oAIo instances across machines (desktop, car SBC, backpack node, etc.) into a unified cluster. Each machine runs its own oAIo stack and registers with a coordinator. Fleet manages discovery, health monitoring, and job dispatch between nodes.
NODE REGISTRATION
Register a remote oAIo instance by URL (e.g., http://100.99.194.124:9000 over Tailscale). Fleet pings the remote /health endpoint to verify connectivity. LAN and Tailscale IPs are allowed; only link-local (169.254.x.x) is blocked for safety.
NODE CARDS
Each registered node shows: name, URL, status (online/offline/error), last heartbeat, capabilities, and hardware summary. The local machine can register itself too.
JOB DISPATCH
Send jobs to remote nodes via the dispatch panel. Select a target node, provide a payload, and Fleet forwards the request. Useful for offloading inference or rendering to a more powerful machine.
FLEET NODES (CONFIG CANVAS)
Two LiteGraph node types: fleet-node (represents a remote machine with status dot) and fleet-dispatch (sends jobs to a selected node). Wire fleet nodes to service nodes to visualize cross-machine routing.

VOLUME MAP
All 17 symlinks under /mnt/oaio/ with their current targets and tiers. Click to change target (repoint a symlink to a different physical path). Supports NVMe, SATA, and RAM (tmpfs via /dev/shm) tiers.
PORT MAP
All registered services with their exposed ports, container names, and network bindings.
CONNECTION MAP
Visualizes how services connect to each other internally (e.g., Open-WebUI → Ollama, Open-WebUI → RVC Proxy → Kokoro-TTS).
TEMPLATES
Save and load named configuration templates. A template snapshots the current services.json + modes.json + paths.json. Useful for switching between different workstation profiles.
API TAB
HOW IT WORKS
Interactive guide showing the oAIo architecture, request flow, and how the frontend communicates with the backend API.
LIVE TOPOLOGY
Auto-refreshing (5s) view of all running services, their connections, and data flow. Shows which services are active and how they route requests.
OPENAPI ENDPOINTS
Full list of every oAIo API endpoint organized by tag (11 tag groups). Each shows method, path, and description. The API is a standard FastAPI app with auto-generated OpenAPI docs at /docs.

Grid density, card sizing, font size. The fluid grid automatically adapts to any screen width.

MODES
Only one mode can be active at a time. Activating a new mode gracefully stops services from the old mode that aren’t in the new one, then starts the new services. Each mode has a VRAM budget — services that don’t fit won’t start.
oLLMo — 11 GB VRAM
LLM pipeline with voice. Ollama (inference) + Open-WebUI (chat + RAG) + Kokoro-TTS (synthesis) + RVC (voice conversion). Talk to the AI, hear it respond in a cloned voice.
oAudio — 5 GB VRAM
Full audio pipeline. Kokoro-TTS (synthesis) + RVC (voice conversion) + F5-TTS (voice cloning). Focused on voice work — training voices, cloning, converting.
comfyui-flex — 18 GB VRAM
Render pipeline. ComfyUI gets nearly the full GPU. For image generation (Flux.1-dev), video, and 3D workflows. Auto-names from the active workflow.

Priority: 1 = highest (last to be killed). Soft-kill (1–2) = warning only. Hard-kill (3–5) = enforcer will stop the container.
Memory mode: “vram” = uses GPU memory. “ram” = uses system RAM only (e.g., Kokoro ONNX).
RESOURCE ENFORCEMENT
The enforcer is a background loop (polls every 5 seconds) that reads GPU VRAM from sysfs and takes action when usage exceeds thresholds.
WARN THRESHOLD (85%)
When VRAM usage crosses 85%, the enforcer logs a warning and adds an entry to the kill log. No services are stopped yet. This is an early alert to reduce load.
HARD THRESHOLD (95%)
When VRAM usage crosses 95%, the enforcer stops the lowest-priority hard-kill service (priority 3–5). Soft-kill services (priority 1–2) are never auto-stopped — they only get warnings. The enforcer kills one service per cycle, re-checks, and repeats if still over threshold.
VRAM CEILING
Override the real VRAM total with a lower virtual limit. Useful for testing if a mode fits on smaller hardware (e.g., set ceiling to 8 GB to simulate an RX 580).
MASTER TOGGLE
Disable enforcement entirely if you want manual control. When disabled, the enforcer loop still runs but takes no action — it only logs.
SYMLINK BUS — /mnt/oaio/
All Docker containers mount volumes through /mnt/oaio/* symlinks, never through direct physical paths. This creates a single abstraction layer between services and storage.
HOW IT WORKS
/mnt/oaio/ollama → /mnt/windows-sata/ollama-models (SATA). To move Ollama models to NVMe, just repoint the symlink: /mnt/oaio/ollama → /mnt/storage/ollama-models. No container restart needed — the container still sees /mnt/oaio/ollama.
STORAGE TIERS
NVMe — fastest, for active models and outputs. SATA — bulk storage for large model collections. RAM (tmpfs) — /dev/shm backed, volatile but fastest possible I/O for scratch data.

CONNECT ANY SERVICE — 3-STEP WORKFLOW
oAIo can integrate any Docker container running on oaio-net. The three-step workflow discovers, probes, and wires new services automatically.
1SCAN DOCKER
Scans the Docker daemon for containers on oaio-net that aren’t already registered as oAIo services. Shows container name, image, status, and exposed ports. If you install new AI software in Docker and connect it to oaio-net, it appears here.
2SCAN API
Probes a registered service’s HTTP endpoints to discover its API capabilities. The scanner checks for:
OpenAPI — /openapi.json, /docs (FastAPI/Swagger).
OpenAI-compat — /v1/models, /v1/chat/completions, /v1/audio/speech, /v1/embeddings, /v1/images/generations.
Gradio — /info (v4), /gradio_api/info (v6) for Gradio-based UIs (RVC, F5-TTS, StyleTTS2).
Generic — common paths like /health, /api/generate, /transcribe.
Results show endpoint count, detected capabilities (chat, tts, embeddings, image_gen, etc.), and the full endpoint list with methods and response codes.
3AUTO WIRE
Generates LiteGraph I/O port definitions from scan results. Maps capabilities to typed ports (chat → string/string, tts → string/audio, image_gen → string/image, etc.). Shows a preview of the generated ports with APPLY / REVERT buttons. Applying updates the service’s node in the CONFIG canvas with the correct I/O ports.
Tip: You can also use scan results as context for manual wiring. Export the scan JSON and use it with any AI assistant to understand what a service can do and how to connect it.

SECURITY
API TOKEN AUTH
Set OAIO_API_TOKEN environment variable to require a Bearer token on all mutating API requests (POST/PUT/DELETE/PATCH). GET requests and the frontend are not gated. Pass the token as Authorization: Bearer <token>.
SECURITY HEADERS
Every response includes: X-Content-Type-Options (nosniff), X-Frame-Options (DENY), Referrer-Policy (no-referrer), Content-Security-Policy (script-src ‘self’ ‘unsafe-inline’ ‘unsafe-eval’, no external scripts).
INPUT SANITIZATION
HTML escaping on all user data rendered in innerHTML (~40 call sites). Path traversal protection on template load/save and file operations. Container name validation via regex (alphanumeric + hyphens only). Fleet SSRF protection blocks link-local IPs (169.254.0.0/16).
NETWORK BINDING
All ports default to 127.0.0.1 (localhost). For remote access, use Tailscale (WireGuard VPN) rather than exposing ports publicly. The oAIo container itself has Docker socket access (necessary for container management).
CONCURRENCY SAFETY
asyncio.Lock() protects 8 config-mutating endpoints against race conditions. Docker SDK calls run via run_in_executor() to avoid blocking the event loop. UUID-based filenames for concurrent /speak requests.
VOICE PIPELINE (oAudio)
The oAudio API at port 8002 provides three endpoints for voice work:
POST /speak
Text → speech. Sends text to Kokoro-TTS for synthesis, then routes the raw audio through RVC for voice conversion. Returns the final audio file. Voice presets: alloy/nova = af_heart, shimmer/echo = af_sky, fable = bf_emma, onyx = am_adam.
POST /convert
Audio → voice conversion. Upload an audio file, it gets converted through an RVC voice model. Use for converting any audio to sound like a target voice.
POST /clone
Reference audio + text → cloned speech. Sends to F5-TTS which generates speech matching the reference voice. For zero-shot voice cloning from a short audio sample.
Adding RVC voices: Drop .pth files into /mnt/oaio/rvc-weights/. They appear automatically in the RVC model list. Index files (.index) go in /mnt/oaio/rvc-indices/.

TROUBLESHOOTING
SERVICE WON’T START
Check if the container exists: docker ps -a | grep <name>. Verify it’s on oaio-net: docker inspect <name> | grep oaio-net. Check logs via LIVE tab or GET /services/{name}/logs. Common cause: another service is using too much VRAM — switch modes or kill conflicting services.
VRAM STUCK / OOM
Hit KILL ALL to stop everything and free VRAM. If VRAM is still showing high after killing services, a process outside Docker may be using it. Check: cat /sys/class/drm/card1/device/mem_info_vram_used. ROCm sometimes needs a few seconds to release memory.
NO VOICE OUTPUT
Ensure both Kokoro-TTS and RVC are running (oLLMo or oAudio mode). Check that RVC voice models exist in /mnt/oaio/rvc-weights/. Open-WebUI routes TTS through the RVC proxy at :8001 — if RVC is down, voice output silently fails.
SCANNER SHOWS NO ENDPOINTS
The service must be running for API scanning. Start it first, wait a few seconds for initialization, then scan. Some Gradio services take 10–30 seconds to load models before their API is ready.
TELEMETRY SHOWS 0% / NaN
VRAM readings come from sysfs at /sys/class/drm/card1/device/. If the oAIo container can’t read these files, telemetry breaks. Verify the DRI device mount in docker-compose.yml. Restart the oaio container if readings were working before.
SYMLINK REPOINT FAILS
Target directory must exist and be accessible from inside the oAIo container. Check that the target path is mounted as a volume in docker-compose.yml. The API validates that the new target is under an allowed prefix.
TIPS & SHORTCUTS
Mode switching — No need to manually kill first. Activating a new mode gracefully stops displaced services, then starts the new ones.
Node click — Click a service node’s title bar in CONFIG to toggle start/stop. The bar turns green when running, red when stopped.
Canvas navigation — Drag to pan, scroll to zoom. Right-click for context menu (add nodes, reset view).
VRAM ceiling — Set a virtual limit in the Enforcement card to simulate smaller GPUs and test if modes fit.
Telemetry toggle — Click the TELEMETRY pill in the top bar to show/hide the real-time gauges strip.
Workflow saves — Snapshot everything (modes, services, paths, settings, graph) in SETTINGS > Workflow Saves. Restore anytime.
Pull models — Download Ollama models from CONFIG > Pull Model. Progress shown in real-time. Models stored at /mnt/oaio/ollama.
Add RVC voices — Drop .pth files into /mnt/oaio/rvc-weights/. No restart needed.
